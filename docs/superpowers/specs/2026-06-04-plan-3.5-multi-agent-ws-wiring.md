# Plan 3.5 — tb-streamer multi-agent HTTP wiring

**Date:** 2026-06-04
**Status:** Draft — ready for user review
**Scope:** Add the missing tb-streamer runtime path that lets users actually start and drive multi-agent sessions. Without this, `--multi-agent-flow` is a documented flag with no working user surface.

---

## 1. Goal

Wire `POST /api/sessions/start` and `POST /api/sessions/:id/input` to the multi-agent code path when `MULTI_AGENT_FLOW=true`. Today these endpoints only know about PTY mode; the Plan 3 work added the agent-client + dependency wiring but never connected it to the actual session-creation flow.

After Plan 3.5 lands:

- A mobile (or any HTTP) client can `POST /api/sessions/start` and receive a usable multi-agent sessionId.
- That client can `POST /api/sessions/:id/input` to send user messages.
- WebSocket subscribers receive the existing `session_update` + `agent_output` + `turn_failure` events as the worker progresses.
- The full end-to-end smoke test (started in earlier session, blocked on this gap) becomes runnable.

Out of scope (deferred to milestone C — see §10):

- Deltas-only signal redesign.
- Workflow-held conversation state.
- External Storage references for large turns.
- Retrofit of structured error codes onto existing PTY endpoints (`docs/plans/structured-error-codes-retrofit.md`).

## 2. Decisions locked during brainstorm

| # | Topic | Decision |
|---|---|---|
| 1 | Mode detection | Process-wide `MULTI_AGENT_FLOW` env var. PTY unreachable when set. |
| 2 | Session-create body | `{}` for new conversation, `{conversationId: string}` for resume. |
| 3 | GET response shape | Symmetric. PTY-specific fields nulled in multi-agent. Multi-agent-specific fields (`stage`, `stalledSinceMs`, `reworkAttempt`) optional. Existing mobile app keeps working. |
| 4 | History source | `ConversationCache.getConversationTail()` + 1.5 MB payload guard returning HTTP 413 with `code: SESSION_HISTORY_FULL`. |
| 5 | turnId source | Server-generates nanoid on each input POST. |
| 6 | Input POST response | 202 Accepted, body `{turnId, status: "queued"}`. |
| 7 | Error shape | New multi-agent endpoints return `{error, code}` with structured machine codes. Existing PTY endpoints unchanged (retrofit deferred). |
| 8 | Session-busy semantics | HTTP 429 + `{code: "SESSION_BUSY", retryAfterMs}` when a turn is already in flight. Mobile retries after the hint. Two-layer defense (tb-streamer check + orchestrator queue). |
| 9 | Resume identity | New sessionId per orchestrator instance. conversationId is the durable identity (the JSONL filename). Mirrors how PTY mode's `resumedFromConversationId` already works. |
| 10 | Resume history | `ConversationCache.getConversationTail(conversationId)` — same path as normal input composition. |

## 3. HTTP API surface

### 3.1 `POST /api/sessions/start`

**Multi-agent mode body:**

```jsonc
{}                              // new conversation
{ "conversationId": "conv_abc" } // resume existing conversation
```

**Response (200):**

```json
{
  "sessionId": "sess_2k4mP9",
  "conversationId": "conv_2k4mP9",
  "status": "running"
}
```

For a new conversation, `conversationId` equals `sessionId`. For a resume, `conversationId` is the supplied existing one, while `sessionId` is freshly generated.

**Errors:**

| Status | Code | Trigger |
|---|---|---|
| 400 | `INVALID_BODY` | Body is not valid JSON or has unexpected fields |
| 404 | `CONVERSATION_NOT_FOUND` | Resume requested with a conversationId that has no JSONL on disk |
| 503 | `TEMPORAL_UNAVAILABLE` | Temporal connect / start fails |
| 500 | `INTERNAL_ERROR` | Unexpected failure |

**Handler logic:**

1. If not in multi-agent mode, fall through to existing PTY logic (unchanged).
2. Parse body. If `conversationId` is supplied, verify JSONL exists at the conversations directory; 404 if not.
3. Generate `sessionId = nanoid()`. For new conversations, `conversationId = sessionId`. For resume, use the supplied id.
4. Call `sessionStore.addManaged({id: sessionId, conversationId, status: "running", ...})`.
5. Call `sessionStore.initAgentSession(sessionId, dedupeCapacity)` to attach the dedupe LRU.
6. Call `agentClient.startSession(sessionId)` to spawn the Temporal workflow.
7. Return 200 with body above.

### 3.2 `POST /api/sessions/:id/input`

**Body:**

```json
{ "text": "user's message" }
```

**Response (202):**

```json
{ "turnId": "turn_8k4mP9", "status": "queued" }
```

**Errors:**

| Status | Code | Trigger |
|---|---|---|
| 400 | `INPUT_REQUIRED` | Missing or empty `text` field |
| 404 | `SESSION_NOT_FOUND` | sessionId not in store |
| 409 | `INVALID_SESSION_STATE` | Session exists but isn't in a state to accept input (e.g., cancelled) |
| 413 | `SESSION_HISTORY_FULL` | Composed `UserInputSignal` exceeds 1.5 MB (75% of Temporal's 2 MB ceiling) |
| 429 | `SESSION_BUSY` | A previous turn is still in flight. Body includes `retryAfterMs` |
| 503 | `TEMPORAL_UNAVAILABLE` | Signal send fails |

**Handler logic:**

1. If not in multi-agent mode, fall through to existing PTY logic.
2. Validate `text` is non-empty.
3. Look up session in `sessionStore`. 404 if not found.
4. **Session-busy check:** if `session.currentTurnId !== null`, return 429 with `{code: "SESSION_BUSY", retryAfterMs: 1000}`.
5. Set `session.currentTurnId = generated turnId`. This is the "lock acquired" marker.
6. Build `conversationHistory` from `ConversationCache.getConversationTail(session.conversationId)` — see §4 for the mapping detail.
7. Compose `UserInputSignal = {turnId, prompt: text, conversationHistory}`.
8. Serialize to JSON, measure byte length. If > 1.5 MB, clear `currentTurnId` and return 413 with `code: SESSION_HISTORY_FULL`.
9. Log a trajectory WARN if appropriate (see §6).
10. Call `agentClient.sendUserInput(sessionId, signal)`.
11. Return 202.

`session.currentTurnId` is cleared by the webhook receiver when `event.stage === "done"` or `event.type === "terminal_failure"` for the matching turn.

### 3.3 GET endpoints (symmetric shape)

`GET /api/sessions` and `GET /api/sessions/:id` already exist for PTY mode. In multi-agent mode:

- `path`, `lastOutput`, `firstMessageText`, `lastMessageText`, `filePath`, etc. — return `null`.
- `stage`, `stalledSinceMs`, `reworkAttempt` — populated when relevant.
- `id`, `status`, `startedAt`, `completedAt`, `sessionName`, `model` — populated normally.
- `conversationId` — populated.

`GET /api/sessions/:id/output` is PTY-specific. In multi-agent mode, return 410 Gone with `code: NOT_APPLICABLE_IN_MULTI_AGENT_MODE`. Existing mobile clients that call it unconditionally will see this; that's acceptable because output-streaming in multi-agent mode happens via WebSocket events, not this endpoint.

## 4. `CachedTail` → `ConversationTurn[]` mapping

`ConversationCache.getConversationTail()` returns a `CachedTail` shape derived from JSONL line semantics. The worker's `UserInputSignal.conversationHistory` is a `ConversationTurn[]` defined in `@threadbase/agent-types`.

The mapping function `mapTailToConversationTurns(tail: CachedTail | null): ConversationTurn[]` MUST:

- Return `[]` if `tail` is null (new conversation).
- Iterate `tail.messages` in stored order (chronological).
- For each cached message, emit one `ConversationTurn` with:
  - `role`: `"user"` or `"assistant"` — direct copy of `cachedMessage.role`. Any other role is a defect; log WARN and skip.
  - `content`: derived from `cachedMessage.content` blocks. **Rules:**
    - Text blocks: concatenate `text` fields with `\n`.
    - `tool_use` blocks: skip in multi-agent mode (the worker doesn't replay tool use; that's the worker's domain).
    - `tool_result` blocks: skip.
    - `thinking` blocks: skip.
  - Skip messages whose `content` is empty after the above transformations. They contribute nothing to the LLM's history.
- **The trailing user message** (the one being sent right now) is NOT included in `conversationHistory` — that's what `UserInputSignal.prompt` is for. Caller is responsible for not double-including it.

This mapping is the riskiest deliverable in Plan 3.5 and MUST ship with a unit test covering:
- Empty cache (returns `[]`).
- User-only turn (one message, role=user).
- Full round-trip (user + assistant, both with text blocks).
- Tool-use turn (assistant message with tool_use block — should produce an empty content message, then be skipped).
- Mixed text + thinking blocks (assistant message — thinking stripped, text kept).
- Unknown role (logs WARN, skips).

## 5. Payload-size guard

Per the panel's binding verdict (see `~/.claude/plans/create-an-agent-or-ethereal-pancake.md`), the guard is **MANDATORY in Plan 3.5**, NOT deferred to milestone C.

**Threshold:** 1.5 MB serialized JSON. This is 75% of Temporal's 2 MB single-payload limit, leaving headroom for the signal envelope (Temporal RPC overhead + the `UserInputSignal` fields beyond `conversationHistory`).

The threshold should be **derived, not asserted** (per the critic's non-blocking note). Implementation should measure observed envelope size on a representative payload during development and either confirm 1.5 MB or adjust to a derived figure.

**Behavior at threshold:**

- Return HTTP 413 with `{error: "Conversation history exceeds payload limit", code: "SESSION_HISTORY_FULL", limitBytes: 1572864, observedBytes: <actual>}`.
- Log WARN with sessionId, observedBytes, conversationId, turnCount.
- The mobile app branches on `SESSION_HISTORY_FULL` to surface: "This conversation is too long. Please start a new conversation."

**Trajectory log (early warning):**

Trigger a WARN log when EITHER:
- The session has reached turn 20, 25, 30, ... (every 5 turns), OR
- The composed `UserInputSignal` is ≥ 500 KB (regardless of turn count — catches fast-growth sessions).

Log includes sessionId, turnCount, observedBytes, percentage of limit consumed. Operators see the trajectory before any user hits the hard wall.

## 6. Concurrent-input race semantics

**The problem:** two POSTs to `/api/sessions/:id/input` land within milliseconds. Without coordination, both would read the same stale `ConversationCache` (missing the most recent assistant turn), compose signals with the same history, and the orchestrator would process both turns thinking each is "the next one" — losing the prior turn's response from the LLM's context.

**The solution (two-layer defense):**

**Layer 1: tb-streamer ingress check.** Per §3.2 step 4, `session.currentTurnId` is set when a POST starts processing and cleared when the corresponding `stage=done` webhook arrives. While set, subsequent POSTs return 429 `SESSION_BUSY` with `retryAfterMs: 1000` (mobile retries after 1s; if the previous turn is still going, gets another 429; eventually gets through).

**Layer 2: orchestrator signal queue.** Per milestone B spec §7.2 (already locked), the orchestrator's signal handler queues signals and processes them one at a time. If a mobile client slips through the 429 check (race between check and set), the orchestrator's queue still serializes turn execution.

**Crash recovery:** If tb-streamer crashes mid-turn (between setting `currentTurnId` and the webhook arriving), the field is lost. On restart, the field is null (in-memory only). The next POST sees `currentTurnId === null` and proceeds. The orchestrator workflow on the worker side may still be processing the lost turn — that's fine, it will complete (or fail) per its own retry policy, and the webhook will arrive eventually (idempotent on the dedupe LRU). The "lost turn" race window is the duration of a tb-streamer restart (~3-5s for a Homebrew respawn). Accepted limitation; document.

**Ingress paths to enumerate (per the critic's risk #3):**

- `POST /api/sessions/:id/input` from mobile app — the canonical path. Subject to the 429 check.
- `POST /api/sessions/:id/input` from internal scripts (smoke-session, etc.) — same path, same 429 check.
- Future: WebSocket message? **Not in Plan 3.5.** If/when added, MUST go through the same `currentTurnId` check.

The 429 check is the only mechanism. There is no separate request-level mutex (no `await acquireLock` blocking the HTTP connection) — the response is immediate, the client retries.

### 7.2 tb-streamer/src/types.ts

`ManagedSession` adds:

```ts
/** Multi-agent: id of the in-flight turn, or null when idle. Cleared by the
    webhook receiver on stage=done or terminal_failure. Optional because
    PTY-mode sessions don't have this concept; undefined in PTY mode, never

if (session && session.currentTurnId === event.turnId) {
  session.currentTurnId = null;
}
```

This is the "release the lock" path matching §3.2's "acquire the lock" set.

## 10. Milestone C — what comes after Plan 3.5

The panel's advisor flagged that embedding full `conversationHistory` in every signal is O(N²) bytes in Temporal Event History and WILL hit Temporal's hard limits (4 MB transaction, 50 MB total history) on long conversations. Plan 3.5's payload guard makes this failure graceful — but it doesn't solve the underlying scaling problem.

**Milestone C (deferred, NOT in Plan 3.5 scope):**

- Redesign `UserInputSignal` to be deltas-only: `{turnId, prompt}` with no `conversationHistory`.
- Move conversation history to workflow state, held by the orchestrator workflow.
- Optionally use Temporal's External Storage / claim-check pattern for large individual turns.
- This is roughly 1-2 weeks of focused work and should be scoped as its own milestone.

The full design analysis (with citations to Temporal docs and industry precedent) lives in the panel verdict at `~/.claude/plans/create-an-agent-or-ethereal-pancake.md` and should be migrated to `tb-multi-agent/docs/ROADMAP.md` as a milestone-C entry when Plan 3.5 commits land.

## 11. Risks and open questions

| Risk | Mitigation |
|---|---|
| 1.5 MB threshold may be wrong | Implementation derives observed envelope size; documents the chosen threshold in `AGENT_PAYLOAD_LIMIT_BYTES` default. |
| Trajectory log noise (every 5 turns starting at 20) may be too chatty for short conversations | Configurable via env var. Defaults can be tuned post-launch based on real observation. |
| `currentTurnId` lost on tb-streamer restart | Accepted; documented in §6 crash-recovery. Future work could persist this to SQLite (cheap, ~5 lines) but unnecessary for Plan 3.5. |
| Mobile app doesn't yet branch on `SESSION_HISTORY_FULL` / `SESSION_BUSY` | Plan 3.5 ships the contract. Mobile-app update is a separate work item. Until that lands, mobile shows the generic 4xx error UI; user can manually start a new conversation. |
| The `mapTailToConversationTurns` mapping might lose data the LLM needs (tool_use, thinking blocks) | Conservative mapping (skip those blocks) keeps the worker's input clean. If the LLM behaves worse than expected, the next milestone can add back tool-use roundtripping. |
